浅谈之-http的状态码以及使用场景

发表于 2018-10-05 | 分类于 前端相关 ， 前端浅谈系列 | | 阅读次数:

前言

众所周知，每一个HTTP响应都会带有一个HTTP状态码（HTTP Status Code），是用来表示HTTP服务器响应状态的代码。它由RFC 2616规范定义的，并得到RFC 2518、RFC 2817、RFC 2295、RFC 2774、RFC 4918等规范扩展。作为web开发者，平时经常看到200、301、302、404、500、503等。但是如果想知道这些状态码，表示的是什么，使用场景在哪里？ 我觉得还是挺有意思的。所以专门花了一些时间，结合自己的知识面，专门整理了一下。
HTTP定义遵循一条规则：所有状态码的第一个数字代表了响应的状态。

  • 1 表示消息
  • 2 表示成功
  • 3 表示重定向
  • 4 表示请求错误
  • 5 表示服务器错误

后面决定采用gulp来重写这个组件：
github 传送门 gulp-staticfy
npm 组件传送门 gulp-staticfy
用gulp 的几个好处就是：

  1. gulp 是基于流的方式，不需要像grunt那样进行那么多的io操作
  2. npm 有一个 phantom 的插件： phantomjs-node，在安装这个npm 插件的时候，不用单独再去安装phantomjs 客户端。而是插件在安装的时候，就会去下载phantomJS。而且整个调用非常方便。可以用 promise， 也可以用 co同步（还有最新的async await语法), 而如果是之前grunt-staticfy 那种处理，就比较麻烦，还要用到 exec 去执行 phantom 的脚本， 非常不友好。
